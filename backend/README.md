# FaSa Backend (Axum + Neon Postgres)

Auth API matching the LoginPage/SignUpPage frontend: signup, login, and a protected `/me` endpoint, using JWT bearer tokens and Argon2 password hashing.

## Important: sqlx compile-time checked queries

This uses `sqlx::query!`/`query_as!` macros, which check your SQL against the real database **at compile time**. That means:

- `DATABASE_URL` must be set (in `.env` or your shell) and reachable *when you run `cargo build`/`cargo run`*, not just at runtime.
- The `users` table must exist already (migrations run at app startup, which is after compilation — so for the very first build, run the SQL in `migrations/0001_create_users.sql` against your Neon DB manually once, *then* `cargo build` will succeed).
- For CI/CD without DB access at build time, install `sqlx-cli` and run `cargo sqlx prepare` to generate a `.sqlx` query cache you commit, then build with `SQLX_OFFLINE=true`.

## Deploying on Render

Render has no native Rust runtime, so this ships as a Docker web service (`Dockerfile` + `render.yaml` included).

1. Generate the sqlx offline query cache locally (required since Render's build container can't reach your DB):
   ```
   cargo install sqlx-cli --no-default-features --features rustls,postgres
   cargo sqlx prepare
   ```
   This creates a `.sqlx/` folder — commit it to git.
2. Run `migrations/0001_create_users.sql` against your real Neon DB once (via Neon's SQL editor or `psql`) before first deploy — migrations run at app startup, which is after the build, but `query!` macros need the table to exist when *you* ran `sqlx prepare` locally.
3. Push to GitHub, then in Render: **New > Blueprint**, point it at the repo. It'll read `render.yaml` and create the service.
4. Render will prompt you for the `sync: false` vars it can't generate itself:
   - `DATABASE_URL` — your Neon **pooled** connection string (hostname contains `-pooler`), with `?sslmode=require`
   - `FRONTEND_ORIGIN` — your deployed frontend's URL, e.g. `https://fasa.vercel.app`
   `JWT_SECRET` is auto-generated by Render via `generateValue: true`.
5. Health check is wired to `/api/health` in `render.yaml`, so Render will know if the container comes up correctly.
6. Once live, update your frontend's `VITE_API_URL` to the Render service URL (something like `https://fasa-backend.onrender.com`) and redeploy the frontend.

Note: Render's free tier spins down after inactivity, so the first request after idle will be slow (cold start + DB reconnect).

## Setup

1. Copy `.env.example` to `.env` and fill in your real Neon `DATABASE_URL` (the pooled connection string from your Neon dashboard, with `?sslmode=require`) and a strong random `JWT_SECRET`.
2. Install the Rust toolchain if you don't have it: https://rustup.rs
3. Run it:
   ```
   cargo run
   ```
   Migrations run automatically on startup (creates the `users` table). Server listens on `PORT` (default 8080).

## Endpoints

- `POST /api/auth/signup` — body: `{ "name", "email", "password" }` → `{ "token", "user" }`
- `POST /api/auth/login` — body: `{ "email", "password" }` → `{ "token", "user" }`
- `GET /api/auth/me` — header: `Authorization: Bearer <token>` → current user
- `GET /api/health` — health check

Errors return `{ "error": "message" }` with an appropriate status code (400 validation, 401 auth, 409 duplicate email).

## Wiring up the frontend

In `LoginPage.jsx` / `SignUpPage.jsx`, replace the `setTimeout` mock in `handleSubmit` with a real fetch, e.g.:

```js
const res = await fetch(`${import.meta.env.VITE_API_URL}/api/auth/login`, {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ email: formData.email, password: formData.password }),
})
const data = await res.json()
if (!res.ok) {
  setErrors({ password: data.error })
  return
}
localStorage.setItem('token', data.token)
navigate('/dashboard')
```

Add `VITE_API_URL=http://localhost:8080` to your frontend's `.env`.

## Notes

- Passwords are hashed with Argon2id (not bcrypt) — stronger default in the Rust ecosystem.
- JWT expiry defaults to 168 hours (7 days), set via `JWT_EXPIRY_HOURS`.
- CORS is locked to `FRONTEND_ORIGIN` — update it for production.
- This does not yet implement refresh tokens, email verification, or password reset — say the word if you want any of those added.
